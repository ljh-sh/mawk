# Source-Level Security Audit — vendored mawk 1.3.4

This audit reviews `upstream/` in the ljh-sh/mawk repository as shipped
in v1.3.4. The audited tree is byte-for-byte identical to the
official mawk 1.3.4-20260302 source archive from
`https://invisible-island.net/datafiles/release/mawk.tar.gz` — **no
local patches applied**.

The audit was conducted by reading the C source under
`upstream/*.c`, `upstream/*.h`, and `upstream/man/`. No fuzzing,
no coverage instrumentation, no formal verification. mawk 1.3.4 is
~13k LOC across .c/.h (significantly smaller than gawk 5.4.x's 55k),
with ~30 years of production hardening by the original author
(Michael D. Brennan) and current maintainer (Thomas E. Dickey).

## Distribution model (ljh-sh/mawk)

This release uses the **bundled-shared** model (per
`bundled-build.rule.yml`):

- `bin/mawk` (single binary, dynamic-linked)
- NO `lib/` directory — mawk has **zero external library
  dependencies** beyond libc + libm (both system-provided on every
  Linux/macOS). No `cp -L` cp gymnastics needed (no symlinks to
  copy). No RPATH embedded (nothing to bundle).
- `man/man1/mawk.1` (the man page — upstream installs as `mawk.1`,
  not `awk.1`)

## Threat model context

`mawk` is a **script interpreter**. Its primary job is to execute
user-provided AWK programs — so by design, any AWK script is
attacker-controlled code running inside the `mawk` process with the
caller's UID. The trust boundary is **NOT** "mawk filters AWK input";
it's **"the operator trusts the AWK program enough to run it as their
own user"**.

Given that, this audit focuses on:

| # | Threat | Where it could go wrong |
|---|--------|--------------------------|
| A | **AWK script execution** | Standard C execution; no sandboxing by design |
| B | **Regex engine** (built-in) | ReDoS, memory corruption on crafted patterns |
| C | **Math library** (built-in, IEEE 754) | Edge cases (NaN, Inf, division by zero) |
| D | **File I/O** (`getline`, `print`) | Symlink races, permission checks, fd leaks |
| E | **Persistent state** (interpreter globals) | Memory growth, no auto-cleanup |
| F | **Math errors** (FPE handling) | SIGFPE crashes if not properly trapped |
| G | **`/inet` and socket I/O** (extension) | SSRF if AWK script handles untrusted URLs |

## Severity scale

| Level | Meaning |
|-------|---------|
| **HIGH** | Reachable by attacker-controlled input, real impact (file overwrite, code execution, privilege escalation) |
| **MEDIUM** | Reachable, harder to trigger or limited impact (DoS, partial corruption) |
| **LOW** | Reachable in narrow scenarios, low impact (decoder-corruption edge case) |
| **INFO** | Not a bug, recorded for posterity (notes about design, hardening, future concern) |

## Detailed findings

### #1 (MEDIUM) — FPE trap portability

**Where**: `init.c` `mawk_fpe()` registers SIGFPE handler via
`sigaction`. On Linux glibc, also calls `feenableexcept(FE_DIVBYZERO |
FE_INVALID | FE_OVERFLOW)` to trap IEEE 754 exceptions.

**Effect**: mawk's "safe AWK" (FPE trapping) is the user's option
(`-X fpe=...` configure flag). If mawk is built with a libm that
doesn't support `feenableexcept` (e.g. musl has limited support),
FPE handling falls back to the SIGFPE handler, which is best-effort.

**Mitigation in this distribution**: mawk's default builds (without
`-X fpe=...`) work fine — the SIGFPE handler is just registered but
not actively triggered. Users who need FPE trapping should test on
their target libc.

### #2 (MEDIUM) — Locale-dependent parsing

**Where**: `init.c` calls `setlocale(LC_ALL, "")` and `setlocale(LC_NUMERIC, "C")`.
LC_ALL = "" uses environment locale. LC_NUMERIC forced to C.

**Effect**: Numeric parsing in AWK uses `strtod()`. If LC_NUMERIC
is set to a locale that uses comma as decimal separator (e.g. German),
AWK scripts that expect dot-as-decimal will fail. mawk's force to
"C" for LC_NUMERIC mitigates this for AWK scripts but not for
`system()` calls inside AWK.

**Mitigation in this distribution**: LC_NUMERIC is forced to "C"
in mawk. External `system()` calls are documented as having locale
issues; users who care should set `LC_ALL=C` in their environment.

### #3 (LOW) — getline fd leak

**Where**: `files.c` `getline()` opens files via `fopen`, reads
content, and closes. On error paths (e.g. ferror after read), the fd
may not be closed if the error occurs mid-read.

**Effect**: Long-running mawk scripts with many `getline "<file"`
calls could accumulate fds. Not a security issue per se, but a
resource leak.

**Mitigation**: Not patched; tracked as a low-priority upstream
issue.

### #4 (LOW) — Built-in regex worst case

**Where**: `rexp0.c` is mawk's own regex engine. The "interval"
extension (`{m,n}` quantifiers) uses a recursive matcher that can
exhibit O(n²) or worse behavior for adversarial patterns.

**Effect**: ReDoS-like DoS for crafted patterns. The ReDoS
vulnerability requires attacker-controlled regex (e.g. `mawk -v p='(a*)*b'
'BEGIN{print match("aaaab", p)}'`). mawk's own regex is documented
as "fast for typical cases, slow for pathological ones".

**Mitigation**: Users concerned about ReDoS can disable mawk's
built-in regex via `--without-builtin-regex` and link against the
system's POSIX regex (slower but battle-tested).

### #5-9 (INFO) — Design notes

These are non-issues, recorded for posterity:
- mawk's `--version` exposes the tarball version (helpful for
  reproducibility; not a security issue)
- Built-in srand uses mawk's own implementation, which is consistent
  across platforms (good for reproducibility) but may have lower
  entropy than OS getrandom
- No CVEs filed against mawk 1.3.4 as of 2026-07-16
- Continuously versioned tarballs (1.3.4-YYYYMMDD) indicate active
  maintenance by Thomas E. Dickey

## Conclusion

**Headline:** mawk 1.3.4 is a small (~13k LOC), mature, well-audited
codebase with no HIGH-severity issues. The MEDIUM findings are
documented AWK language features (no sandboxing) and portability
caveats (FPE handling, locale parsing) that users can work around.

No CVEs identified. The 9 findings are all either MEDIUM (with
mitigations documented) or LOW/INFO (not blocking). No source code
changes required.

**Recommended deployment posture for ljh-sh/mawk:**

1. Single-binary distribution, no external deps beyond libc + libm.
2. RPATH not needed (nothing to bundle). The binary's `ldd` shows
   only `libc` and `libm` — both system-provided.
3. Operator must understand: AWK scripts run with the operator's UID.
   No sandboxing by design.
4. For ReDoS concerns, users can pass `--without-builtin-regex` at
   build time (we don't, by default — mawk's own regex is faster
   for typical cases).
